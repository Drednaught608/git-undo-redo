Undo and redo for Git, the way your editor does it.

Reverse your last Git operation (commit, merge, rebase, reset, branch-switch checkout, whatever it was) with a single git undo, and git redo to put it back. It keeps its own append-only operation log, inspired by Jujutsu, so nothing you undo past is ever lost.

Why it's different

Undo the operation, not just the commit.

Most "git undo" tricks only peel back commits. This reverses whatever you actually did (a bad reset, a wrong rebase, a stray checkout) by reading them from git's own operation record, with the tool's own hops stripped out.

↶

Reverses any operation

Commit, merge, rebase, reset, branch-switch - git undo steps back one real operation. It's branch-aware: every state remembers the branch it was on, so you land back on that branch (recreated if you'd deleted it). It never detaches HEAD.

∞

Append-only, nothing lost

Undo then act, and what you undid past stays in the log behind a prime anchor - you can keep walking back to it. The oplog only ever grows.

⊞

An orthogonal oplog

All tracking lives in its own space inside .git. Your branches and refs stay pristine - nothing leaks into refs/heads or your tags.

✓

Your reflog stays clean

It only writes clearly labeled git-undo: / git-redo: entries and never reads the reflog to navigate - so git reflog stays an honest backstop.

⛉

GC-proof

Every state it has ever seen gets a keep-ref, so git's garbage collector can never reclaim an undone or abandoned commit.

◇

Works everywhere

One bash script, one installer. macOS, Linux, and Windows (Git for Windows or WSL) - the same command on all of them.

A small, obvious surface

Five commands, a handful of flags.

Two verbs you'll use constantly, two for looking around, one to grab a commit's changes. Every command takes -h for help (use -h, not --help: git reserves git <cmd> --help for its own manual pages).

git undo
Reverse your last operation (commit, merge, rebase, reset, branch switch). Global by default.
git undo -l, --local
Undo the last edit on the branch you're currently on, using that branch's own log. Stays on your branch; switch anywhere and undo its last edit without being pulled away. (-g/--global forces the shared-log undo.)
git redo
Re-apply an operation you undid (-l for the current branch's edits).
git undo N · git redo N
Do N steps at once, e.g. git undo 3. Refuses and tells you how many are available if N is too many. Combines with -g/-l.
git undo -s, --staged
Undo, but leave the commit directly above you staged instead of removing its changes - the "committed too early, let me edit and re-commit" flow. Works on git redo too.
git oplog
Show the log, newest first, with @ marking where you are. -g (default) shows the shared log, -l the current branch's edits. -c hides prime anchors (-f, the default, shows them); -i drops you to a point you pick (t toggles the view).
git oplog --reset
Drop the tracked oplog so it rebuilds cleanly from git's reflog (a rebuild, not a wipe - recent history stitches back). Your commits and files are untouched.
git opstatus
Show the current HEAD and how many undo / redo steps remain, for the shared log (-g, default) or the current branch's edits (-l).
git take · git take N
Stage the changes from the commit above your current point without moving - the after-the-fact version of --staged, for when you undid and forgot it. git take N reaches N commits up in one shot. Edit-based; needs a clean tree and a branch.

git config undoredo.default global|local sets the default scope for undo, redo, oplog, and opstatus; git config undoredo.oplog full|compact sets the default oplog view; git config undoredo.color auto|always|never controls color (auto = only to a terminal). Flags always override the config.

How it works

Quietly correct, in its own space.

  1. 1

    Stitch the oplog from the reflog

    On first use the operation log is built from git's reflog (the record of what you did) with the tool's own hops stripped. So even cold, the first undo reverses your last real op.

  2. 2

    Reverse operations safely

    Each op records the branch it was on, so undo lands you back on that branch (recreating it if it was deleted) and moves only that branch's tip. A branch switch is reversed by switching back, never by a corrupting reset, and HEAD is never detached.

  3. 3

    Append-only with prime anchors

    Act after undoing and a prime anchor is appended before your new op, so undo lands where you changed and the path you undid past is preserved.

  4. 4

    Protect everything

    Each state gets a keep-ref so git's garbage collector can never reclaim an undone or abandoned commit.

inside .git/ - kept orthogonal
# tracking, tucked away from your refs
.git/git-undo-redo/timeline   the oplog: sha + kind per op
.git/git-undo-redo/cursor     where HEAD sits
refs/git-undo-redo/keep/…     gc protection

# your space, left alone
refs/heads/main            ✓ untouched
git reflog                 ✓ clean backstop

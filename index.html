Why it's different

The undo you already know, for git.

Most "git undo" tricks only peel back commits. This reverses whatever you actually did - a commit, an amend, a bad reset - the way Ctrl+Z does, by reading git's own reflog. You don't change how you use git; you just type git undo.

↶

Like Ctrl+Z, for git

git undo reverses your last change - a commit, an amend, even a stray git reset - and your working tree follows. git redo puts it back. No new tool and no new workflow to learn: the same two verbs you already reach for in your editor.

⛑

Recover a bad reset --hard

The rescue people actually reach for. git reset --hard HEAD~2 and your commits and files come back with one git undo - because undo restores the recorded state, not just the commit pointer.

↥

Committed too early? Keep the changes

Undo as normal, then git take pulls that commit's changes back into your working tree (unstaged) so you can edit and re-commit - instead of reversing them outright.

⊞

Stays out of your way

It reads git's own reflog and writes nothing outside .git/. Your branches, refs, and tags stay pristine - nothing leaks into refs/heads, and git reflog stays a clean backstop.

∞

Nothing is ever lost

Undo then act, and what you undid past is preserved - you can keep walking back to it. Every state it has seen is also protected from git's garbage collector, so an undone or abandoned commit can't be reclaimed.

◇

Works everywhere

One bash script, one installer. macOS, Linux, and Windows (Git for Windows or WSL) - the same command on all of them.

The everyday surface

A handful of small commands.

Day to day it's git undo, git redo, and git take - bare, no flags - plus git goto to move between branches, and git back / git forward to undo and redo those branch moves. Every command takes -h for help (use -h, not --help: git reserves git <cmd> --help for its own manual pages).

git undo
Reverse your last change - a commit, an amend, even a bad git reset --hard - and your working tree follows. Like Ctrl+Z, but for git.
git redo
Re-apply what you just undid. Like Ctrl+Y - available until you do something new.
git take
Committed too early? Undo, then git take copies that commit's changes back into your files (unstaged) so you can edit and re-commit - instead of reversing them. git take N reaches N entries above you.
git goto
A smarter git switch: change branches - create with -c, jump to a remote, every git switch option works - and your uncommitted work is parked when you leave and restored when you return (it waits where you left it, not on the new branch). No stash/pop, and your git stash stays untouched.
git back
Undo for branch navigation: step back to where you were before your last switch / checkout - one move (git back N jumps N). Separate from git undo on purpose, so undoing an edit never yanks you to another branch.
git forward
Redo for branch navigation: re-do a switch you stepped back from with git back. Together they're Ctrl+Z / Ctrl+Y for your branch history.

Power users: a browsable history and status - git undo N (or git undo all) for several at once, git undo --log / git undo --status / git undo -i to look around or pick a point (the log numbers each row by how far it is from @, so you read the count straight off; all on git redo and git back / git forward too), plus git undo --worktree to step through the working-tree versions git goto parks. These are optional; see How it works and the README. Config keys: git config undoredo.scope edit|global sets the bare-command scope (default edit); undoredo.log full|compact the --log density; undoredo.take unstaged|staged where git take lands; undoredo.color auto|always|never the color. Flags always override the config.

How it works · optional

Quietly correct, in its own space.

You never need any of this to use the tool - the commands above are the whole everyday surface. This is the machinery underneath, for the curious: two durable logs, kept independent.

  1. 1

    Two durable logs from one reflog

    Both logs are projected from git's HEAD reflog - edits per branch, branch switches as navigation - with the tool's own hops stripped, then persisted to disk with each entry's reflog event time. So even cold, the first git undo reverses your most recent edit and git back your last switch - and they keep working after git expires the reflog, or a branch is deleted.

  2. 2

    One move at a time

    An edit undo (git undo) is one reset of your branch; a navigation undo (git back) is one checkout back to where you were (recreating a deleted branch, never detaching unless you stepped to a detached spot). Neither replays the other, so git undo N / git back N is a direct jump. Each log self-heals to where HEAD or the tip actually is, so the two stay independent - git undo never yanks you to another branch.

  3. 3

    Append-only with resume points

    Act after undoing and a resume point (marked ↻) is appended before your new op, so undo lands where you carried on and the path you undid past is preserved.

  4. 4

    Protect everything

    Each state gets a keep-ref so git's garbage collector can never reclaim an undone or abandoned commit.

inside .git/ - kept separate
# tracking, tucked away from your refs
.git/git-undo-redo/local/<branch>/  per-branch edit log + cursor
.git/git-undo-redo/nav/            navigation log + cursor
.git/git-undo-redo/wip/            parked working-tree versions
refs/git-undo-redo/keep/…        gc protection

# your space, left alone
refs/heads/main               ✓ untouched
git reflog                    ✓ clean backstop
